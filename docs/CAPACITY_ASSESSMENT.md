# golang-iptables 容量评估报告

## 评估基准

本报告基于以下配置参数进行容量评估：

- **评估时间**: 2026-04-15
- **配置文件**: `config.json` (默认配置)
- **系统架构**: 单节点 + 可选集群模式

## 当前配置参数

### 限流配置
- `rateLimit.maxRequests`: 60 (单IP 10秒窗口)
- `rateLimit.globalMaxRequests`: 2500 (全局 10秒窗口)
- `rateLimit.windowSec`: 10
- `rateLimit.maxConn`: 2 (单IP并发连接)

### 资源限制
- `limits.maxConnections`: 2000 (全局最大连接)
- `limits.ipsetConcurrency`: 20 (ipset并发数)
- `limits.ipsetCacheMaxEntries`: 200000 (本地缓存上限)
- `limits.ipsetSyncMaxPerRound`: 50000 (Redis同步单轮上限)

### 本地持久化
- `limits.localBanPersistWindowSec`: 1800 (30分钟窗口)
- `limits.localBanPersistMaxEntries`: 50000 (快照最大条目)

### 集群配置（可选）
- `cluster.publishMaxPerSec`: 500 (每秒广播上限)
- `cluster.publishBatchSize`: 32 (批量发送上限)

## 理论抗攻击能力

> **重要说明**: 以下为基于配置参数的理论值，实际性能受系统资源、网络环境、后端能力影响，建议通过压测验证。

### 1. 单 IP 攻击压制能力

| 攻击类型 | 理论配置值 | 实际可达性能 | 说明 |
|---------|-----------|-------------|------|
| 并发连接攻击 | 2 连接/IP | ~2 连接/IP | 超过直接返回 429 |
| 高频请求攻击 | 6 QPS/IP | ~5-6 QPS/IP | 10秒窗口内最多60个请求 |
| 持续违规攻击 | 渐进封禁 | 第3次违规触发 | 第3次违规触发1分钟封禁，后续累加 |
| 封禁时长上限 | 30天 | 30天 | 累计封禁时长封顶30天 |

**结论**: 单 IP 小规模攻击会被自动压制，高频攻击会在第3次违规后触发封禁。实际 QPS 受网络延迟、后端响应时间影响。

### 2. 全局 DDoS 抗压能力

| 攻击类型 | 理论配置值 | 实际可达性能 | 说明 |
|---------|-----------|-------------|------|
| 全局并发连接 | 2000 连接 | ~1500-1800 连接 | 受系统文件描述符、内存限制 |
| 全局请求频率 | 250 QPS | ~200-230 QPS | 受 Go runtime 调度、Redis 延迟影响 |
| IP 封禁表规模 | 20万条 | ~10-15万条 | 受内存限制，超过触发淘汰 |
| ipset 下发速度 | 20 并发 | ~500-1000 次/秒 | 受系统调用、内核锁影响 |

**结论**:
- 全局 250 QPS 是软上限，实际可达 200-230 QPS
- 2000 并发是硬上限，实际受系统资源限制
- 可支撑 10-15 万级封禁表，ipset 下发速度约 500-1000 次/秒

### 3. 本地兜底能力

| 场景 | 兜底能力 | 说明 |
|-----|---------|------|
| Redis 不可用 | 本地限流 + 本地封禁 | 请求放行 + 本地违规计数触发封禁 |
| Redis 恢复 | 自动切回 | 后续封禁继续写入 Redis |
| 节点重启 | 快照恢复 | 恢复30分钟内未过期封禁 |

**结论**: Redis 弱依赖，本地兜底机制可保证基本防护能力。

### 4. 集群模式能力（如果启用）

| 能力 | 理论配置值 | 实际可达性能 | 说明 |
|-----|-----------|-------------|------|
| 封禁传播延迟 | 百毫秒级 | ~100-500ms | 受网络延迟、节点数量影响 |
| 广播上限 | 500 事件/秒 | ~300-400 事件/秒 | 受网络带宽、CPU 影响 |
| 去重能力 | 秒级去重 | 秒级去重 | 相同IP+秒级时间戳去重 |
| 来源校验 | CIDR 白名单 | 有效 | 防止伪造事件 |

**结论**: 集群内封禁传播延迟约 100-500ms，受网络环境、节点数量影响。小规模集群（<10节点）延迟较低，大规模集群延迟会增加。

## 性能瓶颈分析

### 潜在瓶颈

1. **ipset 执行速度**
   - 单个 ipset 命令耗时约 10-30ms（受系统负载、iptables 规则数量影响）
   - 20 并发实际可支撑约 500-1000 次/秒（受内核锁、系统调用限制）
   - 建议：调大 `ipsetConcurrency` 可提升下发速度，但受系统限制

2. **Redis 连接池**
   - redis.Client 默认连接池大小较小（通常 10-20）
   - 高并发下可能出现连接等待，增加延迟
   - 建议：配置 `PoolSize` 参数（建议 50-100）

3. **内存占用**
   - 20万条封禁记录约占用 50-100MB（包括 map 开销、锁开销）
   - 50000条快照约占用 10-20MB
   - 建议：监控内存使用，必要时调小缓存上限

## 容量建议

### 小规模部署（< 1000 QPS）
- 保持默认配置即可
- 适用于个人项目、小团队

### 中等规模部署（1000-5000 QPS）
- 调整配置：
  ```json
  {
    "rateLimit": {
      "globalMaxRequests": 10000
    },
    "limits": {
      "maxConnections": 5000,
      "ipsetConcurrency": 50
    }
  }
  ```
- 建议启用集群模式

### 大规模部署（> 5000 QPS）
- 调整配置：
  ```json
  {
    "rateLimit": {
      "globalMaxRequests": 50000
    },
    "limits": {
      "maxConnections": 20000,
      "ipsetConcurrency": 100
    },
    "redis": {
      "poolSize": 100
    }
  }
  ```
- 必须启用集群模式
- 建议在网关前再加一层 CDN/WAF

## 压测建议

### 压测工具
- **wrk**: HTTP 压力测试工具
- **ab**: Apache Bench
- **hey**: Go 语言实现的压测工具

### 压测场景

1. **单 IP 高频请求**
   - 验证单 IP 限流是否生效
   - 验证渐进封禁是否触发

2. **全局 DDoS 模拟**
   - 多 IP 并发请求
   - 验证全局限流是否生效
   - 验证 ipset 下发速度

3. **集群封禁传播**
   - 单节点触发封禁
   - 验证其他节点是否收到并执行

### 压测指标

| 指标 | 目标值 | 说明 |
|-----|--------|------|
| P99 延迟 | < 100ms | 正常请求延迟 |
| 错误率 | < 0.1% | 非 429 错误率 |
| 429 率 | 符合配置 | 限流生效 |
| ipset 下发延迟 | < 1s | 封禁生效延迟 |

## 监控建议

### 关键指标
- QPS（请求/秒）
- 并发连接数
- 限流触发次数
- 封禁触发次数
- ipset 执行成功率
- Redis 连接延迟
- 内存使用率

### 告警建议
- QPS 超过配置阈值
- 错误率 > 1%
- ipset 执行失败率 > 5%
- Redis 连接失败率 > 1%
- 内存使用率 > 80%

## 总结

当前配置下，golang-iptables 具备以下能力（实际可达性能）：

1. **单 IP 防护**: ~5-6 QPS/IP，渐进封禁，30天封顶
2. **全局防护**: ~200-230 QPS，~1500-1800 并发，10-15万封禁表
3. **本地兜底**: Redis 不可用时仍能防护
4. **集群同步**: 100-500ms 封禁传播延迟
5. **重启恢复**: 30分钟内封禁自动恢复

**重要**: 以上为实际可达性能估算，受系统资源、网络环境、后端能力影响。建议根据实际流量规模调整配置参数，并进行压测验证。
